Teoría aplicada a negocios

La teoría se construye con base en literatura clásica de estructuras de datos y se traduce al lenguaje operativo de NeuroBiz.

1. ¿Qué es una pila y por qué importa en negocios?

Cairó Battistutti y Guardati Bueno (2006) describen la pila como una estructura lineal donde las inserciones y extracciones se realizan sobre un mismo extremo. Esa idea, traducida al contexto de NeuroBiz, significa que el dato más reciente queda en la cima y es el primero en consultarse. Joyanes Aguilar (2014) insiste en que la elección de la estructura condiciona la eficiencia del algoritmo; por eso, antes de programar, conviene identificar la lógica del proceso.

En un historial terapéutico, el terapeuta normalmente no empieza por la primera sesión histórica, sino por la más reciente. El comportamiento esperado es LIFO (Last In, First Out): el último dato que entra es el primero en salir. Weiss (2014) explica que esta organización facilita resolver problemas donde la recencia es crítica, como deshacer acciones, retroceder estados o inspeccionar eventos recientes.

2. Representación y operaciones esenciales

Joyanes Aguilar (2014) presenta las operaciones de una pila como acciones controladas sobre un único extremo. En Python, una lista puede comportarse como pila porque append() agrega en el extremo derecho y pop() retira desde ese mismo extremo; la documentación oficial de Python Software Foundation (2024) confirma ese comportamiento de las listas como secuencias mutables.

Desde la perspectiva de negocio, no basta con saber el nombre de la operación; es necesario entender qué decisión habilita. push registra una nueva sesión, pop retira la más reciente, peek consulta sin eliminar y isEmpty previene errores cuando aún no existe historial cargado. El valor pedagógico está en vincular cada operación con una acción concreta del sistema.

pushInserta en la cimaRegistrar una nueva sesión terapéuticaO(1) amortizado
popExtrae la cimaDeshacer la última carga erróneaO(1)
peekConsulta la cima sin eliminarVer el último reporte antes de atenderO(1)
isEmptyVerifica ausencia de datosEvitar consulta a un expediente sin sesionesO(1)

3. Overflow y underflow como riesgos de negocio

En textos clásicos, el overflow aparece cuando una pila de capacidad fija intenta recibir un nuevo elemento sin espacio disponible; el underflow, cuando se intenta retirar un dato de una pila vacía (Cairó Battistutti & Guardati Bueno, 2006). Aunque la lista de Python crece dinámicamente, en los sistemas de negocio suele existir una política de capacidad, por ejemplo: conservar solo las últimas N sesiones visibles para consulta rápida.

Esto es importante didácticamente: la herramienta técnica puede permitir más datos, pero el proceso organizacional puede imponer límites. Por ello, en esta semana se programará una clase Stack con max_size; así, el estudiante entiende que los errores no solo dependen del lenguaje, sino también de reglas de operación diseñadas por la empresa.

Lectura aplicada: si un terapeuta intenta consultar una pila vacía, el problema no es solo un mensaje de error; implica que no hay evidencia inmediata para dar continuidad al caso. Si se excede la capacidad definida, el sistema debe decidir si rechaza, archiva o rota información según una política explícita.

4. ¿Por qué una pila y no otra estructura?

Weiss (2014) recuerda que una estructura se selecciona por el patrón de acceso dominante. En el caso del historial terapéutico, el patrón dominante es la recencia. No es razonable buscar el dato reciente con una lógica FIFO ni recorrer siempre todo el arreglo. La pila ofrece una representación mental clara: lo último ocurrido está arriba.

Esta claridad conceptual reduce la carga cognitiva en el aula. El estudiante puede recordar la definición y luego aplicar la estructura al caso, que es precisamente la transición de Bloom esperada esta semana. Moreno Pérez (2012) aporta además que la programación debe partir del análisis del problema, no del impulso de escribir código sin modelo previo.

Cuando la pregunta operativa es “¿qué pasó de último?”, la estructura más natural suele ser una pila. Esa intuición de negocio es la puerta de entrada para formalizar el concepto LIFO.

5. Trazado manual del estado de la estructura

Antes de ejecutar Python, conviene representar manualmente la pila. Ese paso permite ver cómo cambia la cima tras cada operación y prepara al estudiante para depurar. Un ejemplo sencillo para NeuroBiz sería: push sesión de lenguaje, push sesión sensorial, push sesión de seguimiento TDAH, pop por carga errónea, push sesión de retroalimentación familiar.                     sesiones_iniciales = [
                        {"fecha": "2026-02-03", "tipo": "Lenguaje", "indice_progreso": 6.8},
                        {"fecha": "2026-02-10", "tipo": "Integración sensorial", "indice_progreso": 7.1},

                    ]

                    historial = []

                    def push_sesion(stack, sesion):
                        stack.append(sesion)
                        print(f"Registrada: {sesion['tipo']} -> tamaño actual {len(stack)}")

                    def pop_sesion(stack):
                        if not stack:
                            print("Underflow: no hay sesiones para retirar")
                            return None
                        sesion = stack.pop()
                        print(f"Retirada: {sesion['tipo']} -> tamaño actual {len(stack)}")
                        return sesion

                    def peek_sesion(stack):
                        if not stack:
                            print("La pila está vacía")
                            return None
                        return stack[-1]

                    for sesion in sesiones_iniciales:
                        push_sesion(historial, sesion)

                    print("
Última sesión visible para el terapeuta:")
                    print(peek_sesion(historial))

                    print("
Se detecta una carga duplicada y se deshace:")
                    pop_sesion(historial)

                    print("
Nueva cima después del ajuste:")
                    print(peek_sesion(historial))

Bloque 2: Clase Stack con capacidad máxima y validaciones colab_stack_class.py 
class Stack:
    def __init__(self, max_size=5):
        self._items = []
        self.max_size = max_size

    def is_empty(self):
        return len(self._items) == 0

    def is_full(self):
        return len(self._items) >= self.max_size

    def push(self, item):
        if self.is_full():
            raise OverflowError("Overflow: se alcanzó la capacidad visible del historial")
        self._items.append(item)

    def pop(self):
        if self.is_empty():
            raise IndexError("Underflow: no existen sesiones para retirar")
        return self._items.pop()

    def peek(self):
        if self.is_empty():
            return None
        return self._items[-1]

    def snapshot(self):
        return list(self._items)

    def __len__(self):
        return len(self._items)

historial = Stack(max_size=3)

for sesion in [
    {"fecha": "2026-02-03", "tipo": "Lenguaje"},
    {"fecha": "2026-02-10", "tipo": "TEA - patrones"},
    {"fecha": "2026-02-17", "tipo": "TDAH - hiperfoco"},
]:
    historial.push(sesion)

print("Estado inicial:", historial.snapshot())
print("Cima actual:", historial.peek())

try:
    historial.push({"fecha": "2026-02-24", "tipo": "Acompañamiento familiar"})
except OverflowError as error:
    print(error)

print("Sesión retirada:", historial.pop())
print("Nueva cima:", historial.peek())
Uso pedagógico: Hace visible la noción de overflow y underflow, mostrando que una restricción de negocio también se programa.
Bloque 3: Pila de acciones para función deshacer (undo) colab_stack_undo.py 
class AgendaUndo:
    def __init__(self):
        self.historial_acciones = []
        self.agenda = {
            "paciente": "Camila",
            "sesion_activa": "Lenguaje",
            "terapeuta": "Laura Pérez",
        }

    def registrar_accion(self, descripcion, cambios):
        self.historial_acciones.append({
            "descripcion": descripcion,
            "cambios": cambios,
        })
        self.agenda.update(cambios)

    def deshacer(self):
        if not self.historial_acciones:
            print("No hay acciones para deshacer")
            return
        ultima = self.historial_acciones.pop()
        print(f"Se deshace: {ultima['descripcion']}")
        # reconstrucción mínima del estado usando una copia base
        self.agenda = {
            "paciente": "Camila",
            "sesion_activa": "Lenguaje",
            "terapeuta": "Laura Pérez",
        }
        for accion in self.historial_acciones:
            self.agenda.update(accion["cambios"])

    def estado(self):
        return {
            "agenda": dict(self.agenda),
            "acciones_pendientes": len(self.historial_acciones),
        }

sistema = AgendaUndo()
sistema.registrar_accion("Cambio de terapeuta", {"terapeuta": "Carlos Rojas"})
sistema.registrar_accion("Cambio de sesión", {"sesion_activa": "Seguimiento TDAH"})

print("Antes de deshacer:")
print(sistema.estado())

sistema.deshacer()
print("Después de deshacer una acción:")
print(sistema.estado())
